18. クローラのナビ抽出（<a href> 以外）

深度 1 以上のクロールで、素の <a href> 以外のナビゲーション （javascript: / onclick / アイコンクラス付き拡張子なしリンク / サーバスクリプト .php リンク）から子ページを辿り、その先の PDF を検出できるかのテスト。

← トップ
対象コード: popup.js の fetchAndParse（クロール時のページ解析）と crawlUrl（フロンティア管理）。
  • jsUrlLiteralRe + routeJsCandidate（popup.js 9091 / 9093 付近）： a[href^="javascript:"] / [onclick] の文字列から クォート済みリテラルを抽出し、ファイル形なら addFileUrl、 それ以外はクロールリンク候補として links に積む（zocbo.com 由来）。
    抽出できるリテラルの形（重要な制約・学んだ落とし穴）: jsUrlLiteralRe がマッチするのは (1) 絶対 URL（http(s)://… や //host/…）、 (2) / で始まるルート相対パス、 (3) / を一切含まない素のファイル名（例 js-target.html） の 3 形のみ。
    crawl/js-target.html のように途中に / を含む サブディレクトリ相対パスはマッチしない（node で実 regex 検証済み）。 そのため本テストの A・B は、ハブと同じ pages/ 直下に置いた 素のファイル名で参照している。
  • crawlFileExtRe のサーバスクリプト除外（popup.js 7973 付近： php\d?|phtml|phps|asp|aspx|jsp|jspx|jspa|cfm|cfml|cgi|fcgi|do|action|ashx|axd|svc|mvc）： classes.php?id=N のようなリンクは「ファイル」ではなく 「次ページへのルート」としてフロンティアに積まれる（drop されない） （yotsuyaotsuka chugaku_kakomon 由来）。
  • detectIconTypeHint + icon-pdf（popup.js 8798 / 8801 / 9038 付近）： 拡張子のない <a class="icon-pdf" href=".../question/">問題PDF</a> は アンカーループ内でアイコンクラスの type ヒントが PDF と判定し、 その URL 自体を深度 0 の PDF 候補として files に追加する （toshin new_kakomon_db 由来）。これはファイル分岐であって 「ページとしてクロールする」分岐ではない点に注意。
確認方法: 拡張機能でこのページを クロール深度 1 以上（ディープスキャン）で Scan してください。
深度 0（このページのみ）では、ハブ自身が直接持つファイル、 および icon-pdf の拡張子なし URL（= それ自体が PDF とみなされる）が検出されます。
深度 1 で、JS / onclick / .php相当 のナビが指す子ページに到達し、 そこにある ../files/* が追加検出されます。
（icon-pdf の crawl/question/ はページとしては辿られない＝中の PDF は個別検出されません。下の C 参照。）

A. javascript: スキームの href

→ JS ターゲットページ（問題一覧）

href は javascript:goPage('crawl-js-target.html')。素の href としては辿れないが、 jsUrlLiteralRe がクォート内の 'crawl-js-target.html'（/ を含まない 素のファイル名・.html）を抽出し、クロールリンク候補にする。 → 子ページ crawl-js-target.html（ハブと同じ pages/ 直下）が深度 1 で解析される。
注意: 旧来の 'crawl/js-target.html' のように / を含む形は jsUrlLiteralRe にマッチしないため、素のファイル名にしてある。

B. onclick ハンドラ（href="#"）

→ onclick ターゲットページ（解答一覧）

href は #（自分自身）だが、onclick 文字列の中の 'crawl-onclick-target.html'（/ を含まない素のファイル名）を jsUrlLiteralRe が抽出 → クロールリンク候補。 → 子ページ crawl-onclick-target.html（ハブと同じ pages/ 直下）が深度 1 で解析される。

C. 拡張子なし icon-pdf リンク（深度 0 で PDF 検出）

問題PDF

href は crawl/question/（末尾スラッシュ・拡張子なし）の素の <a href> （javascript: ではないのでサブディレクトリ相対パスでよい）。通常はナビリンクに見えるが、 class="icon-pdf" を detectIconTypeHint が読み、アンカーループ内で この URL 自体を深度 0 の PDF 候補（typeHint=pdf）として files に追加する （toshin new_kakomon_db と同型。guess 扱いで nonFileUrlRe を回避）。
つまり拡張機能は crawl/question/ を「PDF へのリンク」とみなすだけで、 HTML ページとしてクロール（解析）はしない。 そのため中の sample-2.pdf は個別には検出されない。
本テストサイトはこのディレクトリを crawl/question/index.html として配信するので URL 自体は 404 にならない（解決する）。ただし返るのは HTML であり実体 PDF バイト列ではないため、 ダウンロードの正当性は静的ホストでは保証されない。 本テストの主眼は「拡張子なし URL が深度 0 の PDF として検出されること」。

D. サーバスクリプト .php ナビリンク

学校別ページ（classes.php?id=1）

学校別ページ（classes.html?id=1・静的ホスト用の同等品）

.php についての注意（静的ホストの制約）: classes.php?id=1 は crawlFileExtRe のサーバスクリプト除外により 「ファイル」とは扱われず、クロールフロンティアに積まれる（drop されない）のが 正しい挙動。これは php が allKnownExts に含まれるため かつては classes.php がファイル扱いされてクロールから落ち、学校ページに到達できなかった 回帰（yotsuyaotsuka）の修正点。
ただし GitHub Pages は .php を実行・配信できないため、 実機のライブ検証では classes.php?id=1 自体は 404 になり中身を解析できない。 そこで同じナビ構造を辿れるよう、classes.html?id=1 を等価の到達可能ページとして併置している。 .php リンクは「クローラが enqueue する（落とさない）こと」を文書化するための例であり、 クエリ付き .html 版で実際の到達と PDF 検出を確認できる。

期待される検出

深度 0（このページのみ）で検出されるべき — 2 件:
  • ../files/sample.pdf（ハブ直下の素の a href）。
  • crawl/question/（C の icon-pdf 拡張子なし URL） — detectIconTypeHint が icon-pdf を PDF と判定し、 URL 自体を PDF 候補として深度 0 で追加する。
深度 0 で検出されてはならないもの:
  • A / B / D の子ページ内のファイル（深度 1 以上で初めて辿られる）。
深度 1（ディープスキャン）で追加検出されるべき:
  • crawl-js-target.html 経由 → ../files/sample-1.pdf （javascript: href の素のファイル名を jsUrlLiteralRe で抽出 → enqueue）。
  • crawl-onclick-target.html 経由 → ../files/sample.docx （onclick の素のファイル名を jsUrlLiteralRe で抽出 → enqueue）。 docx は子ページの素の a href なので確実に検出（DOC 系を選択時）。
  • crawl/classes.html?id=1 経由 → ../files/sample.pdf （サーバスクリプト相当のナビが enqueue され到達。sample.pdf は 既に深度 0 で検出済みなので URL 重複（dedup）となり件数は増えない）。
深度 1 でも検出されない（重要）:
  • C の crawl/question/ 内の sample-2.pdf — C の URL はページとしてクロールされない（深度 0 のファイル分岐に入る）ため、 その内側の sample-2.pdf は個別には到達・検出されない。
挙動として確認したいこと（検出件数ではなく経路）:
  • javascript: / onclick 内のクォート済み素のファイル名（A・B）が jsUrlLiteralRe で抽出されフロンティアに積まれること （/ を含むサブディレクトリ相対パスは抽出されない、という制約込み）。
  • icon-pdf の拡張子なし URL（C）が「ナビ」ではなく 「それ自体が PDF」として深度 0 で検出されること （ページとしては辿られない）。
  • classes.php?id=1（D）が crawlFileExtRe でファイル扱いされず enqueue されること （静的ホストでは .php は 404 だが、enqueue されること自体が正しい挙動。 到達と PDF 検出は .html?id=1 版で確認）。
検出されてはならないもの:
  • goPage / location.href などの関数名やパス形でない引数 （'#' 等）はファイルにもクロールリンクにもならない。
