popup.js の Active Capture (Lv2) 検出ブロック（emitTrigger / filenameExtFromText / findRowFilename、5369-5469 行付近）と、 Tokio Marine SPS (u.tmnf.jp) アダプタ（a.download_btn[data-donwloadactionurl] / a.thumbnail_image[...] → postForm、5470-5572 行付近）を再現する DOM 構造です。 実体は page-interceptor.js の activeCaptureArmed() / maybeCaptureBytes / __GAF_ACTIVE_CAPTURE__ ハンドラと連携します。
重要 — この2系統は「クリックして初めてバイトが届く」検出です:
・Scan の時点で拡張機能ができるのは、クリック候補（トリガー項目）の一覧化だけです。実 URL は DOM に存在しません。
・実ファイルのバイトは、ダウンロード / ZIP / 結合 を押した時に拡張機能が各トリガーを 自動クリックし、ページ自身が発行する fetch / XHR / blob を page-interceptor.js が傍受して初めて取得されます （ロケール activeCaptureWarning の説明どおり）。
・SPS の実案件が検出ゼロだった真因は Edge の IE モード（internetexplorertabhost: タブ）に Chromium 拡張の DOM が届かなかったことであり、本静的モックには無関係です。本モックは通常の Chromium タブで動きます。
・本ページのクリックはすべて 無害です（ページ遷移せず、../files のサンプルを取得するか no-op）。

Active Capture Lv2: hrefless ダウンロードトリガー

本物の href を持たない（# / javascript: のみ、または属性なし）操作要素で、 表示テキストが選択中スキャン種別の拡張子付きファイル名になっているもの。拡張機能は実 URL を解決できないため、 __GAF_ACTIVE_CAPTURE__::… というセンチネル項目（activeCaptureTrigger:true、 activeCapture.filenameHint 付き）として一覧化します。

Tokio Marine SPS (u.tmnf.jp) 風 POST ダウンロード

認証必須の保険ポータルを模した構造。ファイルは小さなサムネイルだけが表示され、実バイトは ASP.NET MVC の POST エンドポイント越しにあります。href が存在しないため通常のアンカー走査も Lv2 も拾えず、専用アダプタが postForm 項目を組み立てて通常のダウンロード経路に流します（addUrl(dlUrl, name, true, ext, anchor, postForm)）。

指紋となる目印（ホストゲート不要）: スペルミスの data-donwloadactionurl、 data-gfs-postactionhelper-*、子 img.asyncThumbnailData[th-data-fileId] （th-data-fileType 2=文書 / 1=画像、th-data-extension）、 人間可読名は兄弟の .phInfoThumbnailComment[title]、 反 CSRF トークンは input[name="__RequestVerificationToken"]。

アダプタの仕様メモ: 文書は {name:"attachmentId", value:fileId}（DOM 属性は data-downloadid だが サーバの param は attachmentId）＋トークンを application/x-www-form-urlencoded で POST。 画像は直接バイト経路が無いため /ag/LargeImage/Index へ imageId/direction/dispType/subCount＋トークンを POST し、返却 HTML 内の base64 data: URI を抽出する postForm.inlineImage:true 扱い。 全ファイルが同一エンドポイントへ POST するため、addUrl の URL 重複排除を避ける目的で 無害なクエリ ?_gafid=<fileId> を付与して識別します。

クリック後（ダウンロード / ZIP / 結合 実行時）に初めて起こること: 各トリガーが自動クリックされ、本モックでは ../files/sample.* を fetch して取得します。 SPS 項目は buildItemFetchInit 相当の POST 再生で取得を試みます（本モックの POST は no-op なので実バイトは届かない場合があります。検出＝候補一覧化までを確認してください）。

検出されてはならない:
・「削除」ボタンの行（近くに 重要データ.pdf があっても MUTATE_RE で除外）。
・「申請書.pdf を送信」ボタン（送信 = MUTATE 語を含むため除外）。
・「プロモ動画.mp4」（AV 拡張子 = avExtRe で除外。種別選択にも通常含まれない）。
・本物の href を持つリンクが Lv2 トリガーとして二重検出されること（Lv2 は href 無し要素のみ対象）。

注意: 拡張子付きテキストでも、その拡張子が現在のスキャン種別選択に含まれていない場合は filenameExtFromText が空を返しトリガー化されません。pdf / xlsx / docx / pptx / jpg を有効にして確認してください。
